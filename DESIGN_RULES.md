# DESIGN RULES

## 結論

本ファイルは、`FileUpAPI_AWS` における設計と実装の共通ルールを管理するための基準書です。  
機能仕様や要件は `SPEC.md` で整理し、そこでユーザー承認を得た内容のうち、継続的に参照すべき設計ルールのみを本ファイルへ追記します。

## 目的

- `SPEC.md` の肥大化を防ぐ
- 命名、例外、コメント、責務分割の基準を一元管理する
- 今後の機能設計とクラス設計で参照する前提を固定する

## 本ファイルの更新ルール

- 設計の論点整理や壁打ちは、まず `SPEC.md` で行う
- `SPEC.md` 上でユーザー承認を得た内容のうち、今後も繰り返し参照する設計ルールだけを本ファイルへ反映する
- 一時的な検討メモ、未承認の案、機能固有の仕様は本ファイルに記載しない
- 本ファイルは、実装詳細ではなく「判断基準」を残すことを優先する

## 本ファイルに記載する対象

- 命名規則
- 例外方針
- コメント方針
- レイヤ責務方針
- 共通 DTO や共通レスポンスの設計原則
- 将来の追加実装でも再利用される設計判断

## 本ファイルに記載しない対象

- API ごとの個別仕様
- DynamoDB や S3 の個別設定値
- 画面や CLI の操作手順
- 未承認の案
- 一時的な比較検討メモ

## 今後の設計作業の流れ

1. `SPEC.md` で論点を整理する
2. ユーザーと内容を確認し、採用方針を確定する
3. 確定内容のうち、今後も参照すべき設計ルールを本ファイルへ追記する
4. 本ファイルを前提として、機能設計とクラス設計へ進む

## AWS作業運用方針

### 基本方針

- AWS作業は、可能な限り1セッションで完結する
- AWS作業に入る前に、ローカルで実施可能な確認を先に完了する
- 中断リスクと消し忘れリスクを下げるため、開始時点で終了条件を明確化する

### AWS着手前に必ず実施すること

- `mvn test` と `mvn package` を実行し、ローカルで成立する範囲を先に確認する
- 変更内容に対応するテスト観点（正常系、境界系、異常系、副作用）を明文化する
- 当日のAWS作業の終了条件を明記する
  - 例: API 3本の疎通確認、CloudWatch Logs確認、後片付け完了

### AWS作業中のルール

- 検証対象を最小化し、AWSでしか確認できない項目に集中する
- 作成リソースには識別用タグを付与する
  - 例: `Project=FileUpAPI_AWS`
- 途中で中断する場合でも、削除対象と未削除対象を必ず記録する

### AWS作業終了時のチェック

- 不要リソースを削除したことを確認する
  - 例: Lambda, API Gateway, S3, DynamoDB, 関連ロググループ
- 削除対象を残す場合は、理由と期限を記録する
- 実施結果を `TODO.md` か `SPEC.md` に反映し、次回の再開ポイントを明示する

## 命名規則

### 基本方針

- パッケージ名は小文字のみを使用し、役割単位で分割する
- クラス名は役割が分かる名詞とする
- メソッド名は動詞始まりとし、処理内容が分かる名前にする
- 略語は一般的に通用するもののみ使用する

### パッケージ命名

- `handler`
  - Lambda の入口および外部アダプター層
- `service`
  - 業務処理を扱う層
- `repository`
  - 永続化を扱う層
- `dto`
  - API 入出力を表現する層
- `model`
  - 業務概念または保存モデルを表現する層
- `config`
  - 環境変数や設定値を扱う層
- `validator`
  - 入力値検証を扱う層
- `util`
  - どの特定責務にも属さない純粋な共通処理のみ

### クラス命名

- `Handler` は外部リクエストを受ける入口クラスとする
- `Service` は業務処理を扱うクラスとする
- `Repository` は DynamoDB などの永続化処理を扱うクラスとする
- API 入力 DTO は `*Input` とする
- API 出力 DTO は `*Response` とする
- 永続化モデルは保存単位が分かる命名とし、必要に応じて `*Record` を用いる
- 入力値検証クラスは `*Validator` とする
- Enum は業務概念をそのまま表す命名とする

### 命名上の注意

- `Util` は安易な退避先として使わない
- ドメインや業務文脈を持つ処理は、本来の責務を持つクラスへ配置する
- 意味の薄い短縮語や曖昧な名前は避ける
- S3キー生成のような業務固有ルールを持つ処理は、`util` よりも責務が明確な命名と配置を優先する

### テストメソッド命名

- テストメソッド名は次の形式で統一する
  - `対象メソッド_Check確認観点_with条件`
- `Check` の直後には、何を確認するテストかを明示する
  - 例: `CheckRequiredUserId`, `CheckFormatValidation`, `CheckInitialStatus`
- `with` の後には、入力値や前提条件を明示する
  - 例: `withBlankUserId`, `withInvalidFormat`, `withValidInput`
- 命名でテスト意図が分かることを優先し、曖昧な省略は避ける

## 例外方針

### 基本方針

- 例外は独自例外を用いて意味を明確にする
- 現時点では、HTTP ステータスコードとの対応表までは確定しない
- 先に、例外の分類と送出責務を定義する

### 例外分類

- `ValidationException`
  - 入力値や必須条件の不備を表す
- `ResourceNotFoundException`
  - 取得対象が存在しないことを表す
- `BusinessRuleException`
  - 業務ルール違反や状態不整合を表す
- `SystemException`
  - 外部サービス障害や想定外障害を表す

### 責務分担

- `Handler` は例外を受け取り、外部向けレスポンスへ変換する
- `Validator` は入力値の妥当性を検証し、不正時は `ValidationException` を送出する
- `Service` は業務上の妥当性を検証し、必要に応じて独自例外を送出する
- `Repository` は永続化や外部アクセス失敗時に、必要であれば `SystemException` へ変換する

### 補足

- 例外の意味を HTTP ステータスコードに直接埋め込まない
- エラーコード体系、HTTP ステータス対応、エラーレスポンス形式は別途設計する

## コメント方針

### 基本方針

- コメントは少なめとし、自明な処理には記載しない
- コメントは「何をしているか」ではなく「なぜそうしているか」を補足する
- 変更履歴は Git を正本とし、ソースファイルのヘッダーに履歴欄は持たない

### Javadoc 方針

- クラスには役割が分かる Javadoc を付与する
- `public` / `protected` メソッドには、原則として `@param` `@return` を付与する
- 例外を送出する設計のメソッドには、必要に応じて `@throws` を付与する
- コンストラクタには用途を明記し、必要に応じて `@param` を付与する
  - 例: Lambdaランタイム用、依存組み立て用、テスト注入用
- `package-private` / `private` メソッドでも、責務が誤解されやすい箇所は同形式の Javadoc を付与してよい
- Javadocには「何をするか」に加え、「なぜ必要か」を1行で補足する
- 特に外部公開に近いクラスや責務が誤解されやすいクラスでは Javadoc を優先する

### コメントを許容するケース

- AWS の制約や前提に依存する処理
- 分岐理由や設計判断を明示する必要がある処理
- 将来拡張時に意図が失われやすい箇所

### Javadoc コメント例

クラス:

```java

 * S3のObject Createdイベントを受け取り、受付状態を更新するハンドラー.
 *
 * EventBridge経由で通知されたイベントを処理し、受付状態反映をAPI処理から分離するために使用する。

```

コンストラクタ（用途明記）:

```java

 * Lambdaランタイム用のデフォルトコンストラクタ.

 * 環境変数から設定を読み取り、本番実行に必要な依存を組み立てる。
 */
public RequestStatusUpdateHandler() {

/**
 * 設定値をもとに依存を組み立てるコンストラクタ.

### テストコメント方針

- テストメソッドの直前に、テスト観点を短く記載する
- コメントには「確認項目」を箇条書きで明示し、アサーション意図を一覧化する

## 公式リファレンス確認ルール

### 基本方針

- 採用する技術要素（SDK設定、エンドポイント、URL形式、接続方式）は、実装前に公式リファレンスを確認する
- 推測や一般論だけで設定値を採用しない
- 公式根拠を確認できない場合は「未確認」と明記し、採用判断を保留する

### 運用ルール

- 設計判断を行うときは、根拠となる公式URLを `SPEC.md` または記事メモに残す
- エラー発生時は、まず公式仕様の解釈（Host/Path判定、必須前提、制約）に照らして原因を確認する
- ローカル検証用設定でも、本番想定の方式と整合するかを確認する
- 手順書の変更で実行値（endpoint/region/認証/接続方式）が変わる場合は、同時に実行設定ファイル（例: `env.local.json`）を更新する
- 手順書更新後は、完了報告前に次を必ず突合する
  - 手順書の期待値
  - 実行設定ファイルの実値
  - 設定変更後の再起動実施有無
  - 再実行レスポンスへの反映有無

### 具体例（S3 + LocalStack）

- S3 Presigned URL は、Host/Path の解釈で bucket/key 判定が変わる
- LocalStackのS3接続では、Path-style / Virtual-hosted-styleの公式記述に従って設定する
- 接続方式の誤りは `NoSuchBucket` などの誤判定につながるため、方式を先に固定してから実装する
- 設計書がなくても、テストコード単体で確認範囲が分かることを優先する
- コメントはテスト名と矛盾しない内容にする

// - 更新処理が実質実行されない（Repository呼び出し条件を満たさない）
@Test

}
```

```java

// 確認項目:
// - 戻り値が "IGNORED"

```

### コメントを避けるケース

- コードを読めば分かる逐語的な説明
- 代入や単純な条件分岐の説明
- 一時的な検討メモや運用メモ

### 履歴管理

- 変更履歴は Git で管理する
- ソースファイル内に更新履歴の一覧は持たない

## レイヤ責務方針

### Handler

- Lambda の入口として動作する
- HTTP メソッド、パス、リクエストの受け取りを担当する
- DTO 変換や業務処理呼び出しの制御を行う
- 例外を外部向けレスポンスへ変換する
- 業務ロジックや詳細な入力検証を持ち込まない
- パスパラメータは取り出しまでを担当し、妥当性チェックは `Validator` に委譲する
- パスパラメータ取得は `pathParameters` を優先する
- `path` 文字列からの抽出は暫定フォールバックとしてのみ許容し、`pathParameters` の安定運用が確認でき次第削除する

### Validator

- 入力値の必須チェック、空文字チェック、形式チェックを担当する
- `Handler` から呼び出される前提で使用する
- 業務ルールではなく、入力データとしての妥当性を扱う
- リクエストボディだけでなく、パスパラメータなどの外部入力も検証対象に含める

### Service

- 業務処理の中心とする
- 業務ルールの検証、状態決定、各種部品の組み立てを担当する
- 永続化や外部サービス呼び出しを必要に応じて調停する

### Repository

- DynamoDB などの永続化処理を担当する
- 永続化形式とモデルの変換を担当する
- 業務判断は持たない

### DTO

- API の入出力専用とする
- 業務判断や永続化処理は持たない

### Model

- 業務概念または保存対象のデータ構造を表現する
- 状況に応じて業務モデルと永続化モデルを分けることを許容する

### Config

- 環境変数や設定値の読み込みを担当する

### Util

- 複数レイヤで再利用され、かつ特定責務に属さない純粋な共通処理のみを置く
- ドメイン知識を持つ処理やアプリケーション固有処理は置かない
- 受付API固有のキー生成や命名規則を持つ処理は、将来的に責務が明確な専用部品へ寄せることを許容する
